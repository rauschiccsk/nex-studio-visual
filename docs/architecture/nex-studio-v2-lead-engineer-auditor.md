# NEX Studio v2.0.0 — "AI Agent + Auditor" — Design

> Living design doc, built through the Director↔Dedo consultation (started 2026-06-25).
> **Process:** Director asks about a part → Dedo consults/answers → the final decision is recorded here.
> **Rounds:** R1 (basic functions + key differences vs v1) — **CLOSED 2026-06-26**. R2 — systematic Customer-style walkthrough (Director = Zákazník), project creation → production deploy — **IN PROGRESS**; surfaces + fills gaps across the full lifecycle.
> Supersedes the 5-role serial waterfall (Designer / Customer / Implementer / Auditor / Coordinator).

## Core architecture (one line)
A **AI Agent** (SK UI: **AI Agent**) — one strong senior agent that does the core work itself and dynamically spawns ephemeral helper agents for sub-tasks, holding full context end-to-end — plus an independent **Auditor** (the verifier: release-acceptance + adversarial spot-checks) + a configurable **autonomy dial**. Replaces the 5-role serial pipeline. Rationale: dogfood metrics showed the pipeline was ~1.1× human (Programmer 0.6× = slower), 100+ reworks, 2d4h Director-wait — over-engineered for delivered value, especially as models strengthen.

## Decision log

### D1 — What STAYS (only the build engine changes)
Project creation, the version/work hierarchy, and metrics **stay**. More broadly, **only the build ENGINE (the agent pipeline) changes** — the surrounding infrastructure stays and adapts: cockpit, KB integration, UAT provisioning, deploy machinery, multi-module, settings, auth. (Each confirmed part-by-part below.)

### D2 — The 5 agent roles → AI Agent + Auditor
The 5-role serial structure (Designer→Customer→Implementer→Auditor→Coordinator, with handoffs) is **removed**. It collapses into:
- **AI Agent (AI Agent)** — one strong senior agent that does design + implementation + self-verification with full context, like Dedo. It does the core/hard work itself and **dynamically spawns ephemeral helper agents** for parallel/bulk sub-tasks, then integrates the results. It replaces the WHOLE 5-member team — it is NOT a renamed Coordinator. The Coordinator was a *dispatcher* (routed work between fixed roles, carried the "papers"); the AI Agent DOES the work and only pulls in *temporary* helpers on demand. Only the Coordinator's *Director-facing* part survives (reporting status + asking for approval), now done by the AI Agent itself.
- **Auditor** — a SEPARATE, independent agent that verifies the AI Agent's work before release (the verifier). Invoked at high-value points only: the release-acceptance check (run the app, confirm it does what the spec promised) + adversarial spot-checks on risky parts (security, money, core contract). **Stays independent from the AI Agent** (it is NOT one of its helpers — it checks from outside) because no agent can fully audit itself (blind-spot safeguard). See D5 for the targeted-vs-old-per-task distinction.

Note: "agents" do not disappear — the 5 standing ROLES with serial handoffs do. v2.0.0 has the AI Agent (the doer, with on-demand helpers) + the independent Auditor (the checker).

### D3 — Pipeline phases stay VISIBLE (transparency preserved)
**DECIDED (Director): the BUILD pipeline = 4 phases — Príprava → Návrh → Programovanie → Verifikácia** (Director, 2026-06-26; Plán merged into Návrh as its last part = the task plan; the old "Vydanie" removed — deployment is now separate & per-customer, see D6). Príprava ends with the **Špecifikácia, whose approval by the Manažér is ALWAYS mandatory (independent of the autonomy dial)**. (+ Hotovo, terminal). "Návrh" consolidates scope/API/BE/FE design; "Verifikácia" is the **Auditor's** phase (consolidates the old "Kontrola zákazníkom" + "Audit"). Sufficient transparency without faking serial sub-phases. (The AI Agent MAY report finer live detail within a phase if useful, but the visible map is these 4.)
- **Meaning shifts:** the display shows "which stage the **AI Agent** is in" (its progress), not "which agent is active."
- **Approval points (NOT "gates"):** between phases the Director can approve before continuing — these are **schvaľovacie body**, driven by the **autonomy dial** (Director chooses where to approve: release-only, key decisions, or full-auto). Distinct from the Auditor (the independent verifier, not an approval stop).

### D4 — Name of the doer agent → AI Agent
**AI Agent** — used in both UI and code (an international term). The doer/builder; its sidebar tab is the terminal where the Manažér talks to it (a Claude Code session). It still internally spawns ephemeral helpers, but those are NOT standing agents, so the simple title fits. Naming history: Lead Engineer → Developer / Vývojár → **AI Agent** (Director, 2026-06-26). Rejected along the way: Builder/Staviteľ, Maker/Tvorca, Craftsman/Majster (construction metaphors, not IT), Team Leader/Tímlíder (leans management). (The build board/tab is separately named **"Vývoj"** — see the sidebar line.)

### D5 — Name of the verifier agent → Auditor
The independent verifier is named **Auditor** (works in both EN and SK — SK form *Audítor*; familiar from the old role). **IMPORTANT — same name, leaner job:** the OLD Auditor checked **every task** (per-task) — the expensive, over-engineered part we dropped. The NEW Auditor is **targeted**: the release-acceptance check at the end + adversarial spot-checks on risky parts only — NOT per-task. (This was previously called "the Gate"; "Gate" is retired to avoid clashing with the phase approval-points.) Clean terminology: **Auditor** = independent verifier; **schvaľovacie body** = the Director's approval stops (autonomy dial).

### D6 — Deploy & Customers model (Director, 2026-06-26)
**UAT and PROD are PER-CUSTOMER, not universal** — each customer has its own instance + its own data (matches the proven instance-per-customer model: ANDROS / ICC / MÁGERSTAV for nex-inbox, `/opt/customers/<slug>/`).
- **The build pipeline (Príprava→Návrh→Programovanie→Verifikácia) is customer-AGNOSTIC** — it produces + verifies a *version*. **Deployment is OUT of the pipeline** (separate, per-customer).
- **"Zákazníci" (Customers) — a per-project registry.** Each project records which customers use the app, **added via a form** (name, slug, subdomain, integrations, secrets; deploy target = their own UAT + PROD instance/DB/data). Productizes today's `onboard-customer.sh` + auto-UAT-provisioning, in the UI. **Uniform structure: internal apps are treated identically — the "customer" is simply ICC s.r.o.** (no special internal case).
- **New sidebar tabs "UAT" and "PROD"** (project-scoped) — each lists the project's customers + which version each runs (version × customer matrix). The Manažér triggers per-customer: **UAT** = deploy a version to customer X's UAT (their test instance + test data) → test there; **PROD** = after accepting in customer X's UAT, promote to their PROD. **Per-customer acceptance gate (never bypassed).** First PROD deploy of a project = **v1.0.0**. Different customers may run different versions (ANDROS PROD v1.0.0, ICC v1.1.0).
- Deploy = automated per-customer instance provisioning. Verifikácia verifies the app internally (customer-agnostic, with its own test fixtures); the per-customer UAT/PROD is a separate deploy to the customer's own instance.
- **Deploy flow + data (Director, 2026-06-26):** Manažér opens the **UAT** (or **PROD**) tab → picks a **customer** + a **verified version** → **"Nasadiť"** → the system auto-provisions/updates that customer's instance with that version (its own DB, subdomain, integrations, on its own URL). **Data:** a NEW project/customer = a **fresh, empty install** (no seed/test data) — data accumulates through real use (in UAT you test by *using* it; in PROD real data grows). **Every version update to an EXISTING instance MUST PRESERVE the accumulated data** — update code + run schema migrations; **never wipe data or rotate secrets.** First deploy = empty; every later deploy = data-preserving. (Learned once on the inbox UAT redeploy — data/secrets must survive.) (Supersedes the part-by-part "Deploy" + "UAT provisioning" lines below.)
- **UAT acceptance (Director, 2026-06-26):** the **project manager tests** the customer's UAT (on its URL) — a human acceptance (the Auditor's automatic Verifikácia already passed before deploy). In the UAT tab, per customer: a link to the UAT URL + an **"Akceptovať"** button → on click, the version is marked **accepted-for-PROD for that customer**, which **opens the PROD deploy** for them. Per-customer + independent; logged (who / when / version / customer). **No PROD deploy without UAT acceptance** — the acceptance gate, never bypassed. The customer MAY get UAT access to try it too, but the **manager performs the acceptance.**

## AI Agent — the concept (elaborates D2)
The **AI Agent (AI Agent)** owns and delivers the whole build end-to-end with full context — "Dedo, productized." It is **not literally one person**: it does the core/hard work itself and **dynamically creates temporary helper agents** for sub-tasks (parallel/bulk work), divides the work, and integrates the results — exactly how Opus/ultracode operates.
- **Full context, no handoffs:** keeps one warm context the whole way (Príprava→Návrh→Programovanie; deployment is separate & per-customer). It remembers WHY it designed something because it designed it — no re-reading, no handoff drift (vs the old role relay).
- **Dynamic, on-demand team:** helpers are **ephemeral** (spawned for a specific sub-task, return a result, disappear) — NOT standing roles. Small task → no helpers (does it alone); large task → spawns and directs them. Scales to need.
- **Does:** preparation, design, task-plan, coding (with continuous self-checking), and directing/integrating its helpers.
- **Does NOT:** the final independent verification — that is the **Auditor** (separate agent, OUTSIDE the AI Agent's team), because no agent can fully audit itself. The AI Agent is not its own judge.
- **Not a script:** an intelligent agent that decides its own steps, within the phases + autonomy-dial approval points.

## Part-by-part decisions (filled through the consultation)
- **Project creation** — KEEP (D1). **Project types / archetypes (Director, 2026-06-26 — IN DISCUSSION):** drop the old single/multi choice (D7); instead model a project as **one backend + one-or-more frontend "surfaces"**, and the type = a preset composition (scaffold template) the AI Agent uses to start. Surfaces: app-FE · public-web · eshop · mobile. Types:
  1. **Štandardný** = BE + app-FE (NEX Ledger/Inbox).
  2. **Webový** = BE + admin-FE + public web. Key: a *managed/monitored* site — NOT a throwaway static gen (no rockart-web style); the admin-FE is where you configure the site + see its metrics.
  3. **Eshop** = BE + admin-FE + eshop storefront + **bidirectional integration with an information system (IS)** — the real complexity.
  4. **Mobil** = a mobile-app surface addable to any project (shares the BE).
  - **DECIDED (Director, 2026-06-26) — MERGED:** Web and Eshop are ONE **"Web" archetype** = BE + admin-FE + public site; "eshop" = turning ON commerce features (cart/checkout/payments) + bidirectional IS-integration. Same shape → DRY; nex-shared holds the web-platform; commerce is an add-on capability. The **eshop↔IS bidirectional integration is the real complexity** (design carefully). Final type list: **(1) Štandardný** (BE + app-FE), **(2) Web** (BE + admin-FE + public site; optional eshop/commerce + IS-integration), **(+) Mobil** = a mobile surface addable to any.
  - **Mobil = biggest unknown → dedicated design round:** mobile toolchain (RN/Flutter/Expo) + how NEX Studio builds & TESTS a mobile app (emulator/device) + how the Auditor verifies it.
  - Archetypes = scaffold templates; nex-shared holds shared solutions across surfaces.
  - **R2 create flow (2026-06-26):** human operator role renamed **Director → "Manažér projektu" (Project Manager)** — propagates across the UI (approvals, "čaká na manažéra", autonomy-dial owner). **First step (Standard project):** Manažér → Projekty → "Nový projekt" → enter name (→ slug) + pick type **Štandardný (BE + app-FE)** (+ short description, owner for notifications) → submit. Required before a build can start (it is the brief).

On save: the version record is created + the spec saved to `docs/specs/versions/v<N>/customer-requirements.md`. **No autopilot.** After saving the **Zadanie**, the Manažér clicks **"Spustiť tvorbu špecifikácie"** → the system auto-activates the 👨‍💻 AI Agent tab + injects an init prompt (*"Načítaj zadanie a začni prípravu špecifikácie"*) → the **Príprava phase = an interactive terminal dialogue** (like a Director↔Dedo session): the AI Agent reads the Zadanie, systematizes the requirements, **asks the Manažér clarifying questions on every unclear / under-thought point — NO design until every detail is understood**, and **proactively proposes improvements** (features / UX / quality). The Manažér reads it there and **approves via a "Schváliť špecifikáciu" button** in the Príprava tab. **Approval is ALWAYS mandatory (independent of the autonomy dial).** Only then Návrh proceeds (the autonomy dial governs the later phases). **Terminology:** Zadanie = the assignment/brief (amateur input; file `customer-requirements.md`); Špecifikácia = the professional spec from the dialogue (≈ development-spec). **DECIDED (Director, 2026-06-26):** (1) first version = **`v0.1.0`** (base / initial dev version); it bumps to **`v1.0.0` at the first production Deploy**. - **Autonomy dial** (SK UI name: **"Miera autonómie"** — confirmed; **configured in Nastavenia / Settings**) — a Director setting controlling how much the AI Agent does on its own vs how often it stops at a **schvaľovací bod** for approval. The schvaľovacie body are the potential stops; the dial decides which actually halt. **Fast-fix = dial at full-auto.** Solves the old forced-gate problem (the 2d4h Director-wait). - **Verification — the Auditor (TWO touchpoints, Director 2026-06-25):**
  - **Intensity scales with the Miera autonómie / Director's presence (Director, 2026-06-26):** low autonomy (Director in the loop) → Auditor light/minimal (the Manažér + self-check + tests ARE the audit, like Dedo); high autonomy (unsupervised) → Auditor full (the Manažér's independent proxy — the eyes the Manažér would otherwise be). The Auditor exists precisely to enable UNSUPERVISED builds; Dedo-here works auditorless only because the Director is always present.
  - **When the Auditor finds a problem (Director, 2026-06-26):** (i) *implementation problem* (bug / code doesn't match spec / behavioral failure) → the **AI Agent fixes** → Auditor re-verifies → bounded loop (~5 attempts, configurable); if still unfixable (or beyond a code fix) → **STOP + escalate to the Manažér**. (ii) *spec/design hole* (missing/ambiguous info) → **escalate to the Manažér directly** (clarify → revise Špecifikácia/Návrh). The Auditor only FINDS/VERIFIES; the AI Agent FIXES (independence preserved).
  - **(a) Upfront spec/design review** (after Návrh, before the build commits to coding) — the Auditor independently scans the brief (`customer-requirements.md`) + the AI Agent's design for holes / ambiguities / contradictions = **the OLD Customer agent's Gate-E function, now done by the independent Auditor**. PLUS the AI Agent raises its own clarification questions during Príprava/Návrh (it can't build what isn't specified). Both surface at the **schvaľovací bod after Návrh**: the Director fixes/clarifies the brief or approves to proceed. (Replaces the old /dialogue page — inline in the flow now.)
  - **(b) End verification** at the Verifikácia phase — release-acceptance (run the app, confirm it does what the brief promised) + adversarial spot-checks on risky parts. NOT per-task.
  - Depth is **dial-able** (autonomy dial): full independent upfront review for important/regulated projects; lighter (just the AI Agent's questions) for quick ones.
  - Net: the **Auditor = independent conscience at both ends** — upfront "is the plan sound?" + end "did it deliver?". (Updates CLAUDE.md §2.5 — the upfront pillar is now the Auditor's early review, not a standing Customer role.)
- **Fast-fix lane ("Rýchla oprava") — KEEP, simplified (Director, 2026-06-25).** For small fixes that don't warrant a full version build (no Zadanie/Špecifikácia, no full build pipeline). Flow: describe the fix (a **directive**) → **auto-create a patch version** (vX.Y.Z+1) → the **AI Agent** takes the directive (which IS the brief) and fixes it directly on a **short path** (skips the heavy Návrh/Plán) → **light Auditor check** (fix works + no regression — focused, NOT the full release oracle) → **auto-deploy** → **autonomous** (zero mid-flight approvals by default; dial-able to require approval for sensitive fixes). Outside the build pipeline — its own minimal lane (Oprava → quick verify → Deploy). Fits v2.0.0 better than today (one AI Agent instead of 5-role routing). Proven today: nex-ledger fast-fix ~18 min, zero approvals.
