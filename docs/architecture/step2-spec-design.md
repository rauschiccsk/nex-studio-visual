## OPRAVY Z ADVERSARIÁLNEJ KONTROLY (minor — MUSIA sa zapracovať)

Verdikt kritika: **ready**. Štyri spresnenia:
- **[minor]** The design asserts the spec body never lands in the DB, but every AGENT TURN in a conversation build records payload.report = extract_report_body(stdout) (orchestrator.py:2341, pipeline_status.py:491). If the AI partner pastes the full/large spec text into its conversation REPLY (natural chat behaviour: 'here is the current draft…'), that reply's payload.report becomes a stale shadow copy of spec content in the append-only log. It is NOT read by the reader/build/approval (those read the one .md on disk), so it does not break the single-source-of-truth for what gets USED — but it is a de-facto second copy of the text, contradicting the design's absolute 'ŽIADNY DB text stĺpec pre telo spec' framing.
  - **Oprava:** Keep the invariant precise: the spec's single source of truth is the on-disk .md; payload.report is the transcript of the turn, not an authoritative spec store. In _conversation_directive add one line instructing the partner NOT to paste the whole specification.md into chat replies (point to the file / say 'aktualizoval som specification.md'), so the transcript stays a summary and the file stays the only full copy. No code gate needed — a directive nudge is enough and matches the honest-by-construction style.
- **[minor]** The design says SpecMarkdown {body} bundles the 'prose prose-sm dark:prose-invert' wrapper and re-points BOTH ConversationThread and PhaseArtifact. But the wrappers are NOT identical: ConversationThread wraps in 'prose prose-sm dark:prose-invert max-w-none text-sm text-[var(--color-text-primary)]' with NO padding (ConversationThread.tsx:103 and again :113 for the question block), while PhaseArtifact adds 'px-4 py-3' (PhaseArtifact.tsx:107). Only the inner MARKDOWN_COMPONENTS + ReactMarkdown+remarkGfm is byte-identical. If SpecMarkdown hard-codes px-4 py-3, chat bubbles gain unwanted padding (visual regression) and the question sub-block (:113-117) is a SECOND call site the design's file-list does not mention.
  - **Oprava:** Scope SpecMarkdown to the DUPLICATED part only: the ReactMarkdown+remarkGfm+MARKDOWN_COMPONENTS+CodeBlock renderer, taking {body} and an optional className (or no outer padding at all). Let each consumer keep its own prose wrapper/padding. Explicitly re-point THREE call sites in the FE: ConversationThread body (:104) AND its question block (:114), plus PhaseArtifact (:108). Task 6's detail must name the question sub-block so it is not missed.
- **[minor]** SpecifikaciaPage badge 'Schválená/Rozpracované' is specified as derived from 'presence of kind=approval for the version', optionally via usePipelineWs(versionId) board.recent_messages. But board.recent_messages is capped (limit=50, getPipelineBoardApi default) and is the RECENT tail — on a long conversation the approval message can scroll out of the 50-message window, so the badge would silently flip back to 'Rozpracované' after approval. The design does not pin a durable source for the approval fact.
  - **Oprava:** Derive the badge from a durable signal, not the truncated recent tail: either (a) read board.available_actions — once approved, the conversation settles and the approve_spec action's presence/absence is a stable proxy, or (b) add a tiny GET that answers 'is there a kind=approval message for this version' (a count, not the tail). Simplest: reuse the existing listPipelineMessagesApi with a kind/stage filter if available, else fetch board and check state — but do NOT rely on the 50-item recent_messages window as the authoritative approval indicator.
- **[minor]** Minor redundancy / correctness nuance: NewVersionPage already computes 'description: zadanie.trim() || undefined' (NewVersionPage.tsx:114), so an empty Zadanie is already safe for the version row. The design's task 9 must ONLY (a) drop the validate() check at :97 and (b) guard the writeZadanie call at :118 with if(zadanie.trim()). It must NOT touch createVersion/description handling. If the Implementer over-reaches and writes an empty customer-requirements.md when Zadanie is blank, the directive's 'read customer-requirements.md IF EXISTS' would find an empty file and the partner may treat 'exists-but-empty' inconsistently.
  - **Oprava:** Task 9: change exactly two lines — remove the not-empty branch in validate() (:97) and wrap writeZadanie in if(zadanie.trim()) (:118). Confirm no empty customer-requirements.md is created for a blank Zadanie (skip the write entirely), so 'IF EXISTS' in the directive is a clean present/absent test, never present-but-empty.
