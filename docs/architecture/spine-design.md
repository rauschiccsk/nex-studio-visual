## Po ľudsky (pre Manažéra projektu)

"Postaviť chrbticu" (STEP 1) znamená vymeniť staré 4-fázové automatické riadenie stavieb za obyčajný živý rozhovor 1:1 medzi tebou (Manažérom) a AI partnerom — presne tak, ako komunikuješ s Dedom. Doteraz appka viedla stavbu ako pevný automat cez fázy (Príprava → Návrh → Programovanie → Verifikácia) s oddelenými obrazovkami, rozhodovacími kartami a tlačidlami. Namiesto toho postavíme jednu novú obrazovku "Riadiace centrum", kde je uprostred konverzácia (to je tá chrbtica), vpravo miesto pre "Plán úloh" (zatiaľ len placeholder), hore lišta fáz a po ruke čestný stav (Voľný / Pracuje / Čaká na teba / Pozastavené). Celá overená technika pod kapotou zostáva nedotknutá — engine, ktorý posiela prompty partnerovi a streamuje odpoveď, jediný zapisovač do append-only denníka, WebSocket živé aktualizácie, relay (tvoja správa poslaná počas kroku doletí, len čo krok dokončí), autentifikácia aj deploy. Nič sa nerefaktoruje — staré komponenty sa mažú a nahrádzajú. Pridáme jedno nové nastavenie "token-stop" (zastav stavbu po X miliónoch tokenov; 0 = bez zastavenia), ktoré naozaj pozastaví beh na hranici úlohy a pošle ti Telegram, keď si preč. Beží to celé vedľa existujúcich v2 PROD projektov bez toho, aby sa ich čokoľvek dotklo — nový režim sa vyberá pridaným príznakom, staré stavby idú po starom.

## Vyriešené stavebné voľby (partner rozhodol, Manažér potvrdzuje)
- **MODE_TOGGLE** — Ako má appka rozlíšiť nový rozhovorový režim od starých fázových stavieb, aby bežali vedľa seba a nedotkli sa existujúcich projektov?
  - Voľba: **Pridať nový nepovinný príznak 'mode' (prázdny = po starom).**
  - Prečo: Je to najčistejšie oddelenie nového od starého: prázdna hodnota znamená, že každý existujúci v2 PROD projekt ide presne po starom (nič sa nemigruje, nič sa nedotkne). flow_type sa používa hlboko vo fázovom kóde, ktorý zámerne nechávame na pokoji — prepísať ho by tam prepašovalo rozhovorovú logiku. Odvodenie z fázy naráža na databázové obmedzenie povolených hodnôt. Nový príznak je jednoduchý prídavok s triviálnou migráciou.
- **TOKEN_STOP_SCOPE** — Kde má žiť nastavenie 'token-stop' (zastav po X miliónoch tokenov) a na akej úrovni?
  - Voľba: **Jedno GLOBÁLNE nastavenie (možnosť prepisu per projekt/stavba necháme pripravenú, ale nezostavenú).**
  - Prečo: REDESIGN §9 ho výslovne umiestňuje do Nastavenia → Systém a jeho účel je sledovať Anthropic token-politiku centrálne — presne tá runtime-meniteľná, cache-podložená forma bez migrácie. Globálne = nula zmien v schéme a najbezpečnejšie spolužitie (default 0 = doterajšie správanie bez zastavenia pre každý existujúci projekt). Per-projekt/per-stavba prepis vieme kedykoľvek doplniť overeným vzorom, ale stavať ho teraz je špekulatívne.
- **SALVAGE_VS_FRESH** — Máme skopírovať overené UI komponenty starého rozhovoru do nových, alebo napísať konverzačné UI od nuly?
  - Voľba: **Skopírovať do components/riadiace/ a originály zmazať.**
  - Prečo: Starý transcript už vykresľuje správy ako bubliny v štýle Dedo↔Zoltán s čestným zobrazením otázok/reportov, ktoré sa ladili viacero CR-iek, a input box už hovorí relay/deferred protokolom. Použiť ich na mieste sa nedá — staré stránky/komponenty sú CUT (mažú sa) a plán zakazuje nechať CUT kód, aby sa okolo neho refaktorovalo. Písať od nuly zahodí overenú, ťažko získanú prácu. Kopíruj-a-zmaž zachová hodnotu a rešpektuje CUT pravidlo.

## OPRAVY Z ADVERSARIÁLNEJ KONTROLY — MUSIA sa zapracovať do úloh
Verdikt kritika: **needs-revision**. Doplniť do príslušných úloh:
- **[major]** The design keeps the runner's relay-drain loop for conversation builds and claims drain_relay_turn (orchestrator.py:2110) is 'reused UNCHANGED'. But drain_relay_turn hardcodes a call to run_dispatch at orchestrator.py:2134 — it does NOT branch on mode. So an in-flight Manazer message on a conversation build (enqueued during a turn, then drained after it settles) is dispatched through the PHASE AUTOMATON (run_dispatch), not run_conversation_turn. For a conversation build the drained turn runs run_dispatch with stage='priprava' + a directive: on a gate_report reply it hits _persist_priprava_spec (orchestrator.py:3831-3832) and _settle_phase_boundary (orchestrator.py:3846) — exactly the phase-automaton settle logic the spine is meant to replace, leaking spec-persistence/phase-advance semantics into the conversation and violating the 'REPLACE not patch' path-1 rule. The settled-build relay path is fine (apply_action ask/answer -> _begin_dispatch -> schedule_dispatch -> _run, which the design DOES branch on mode), but the in-flight-drain path is not. This is reachable any time the Manazer types while the partner is mid-turn.
  - **Oprava:** Make drain_relay_turn mode-aware: after _begin_dispatch, read the state's mode and call run_conversation_turn (with the directive) for a conversation build, run_dispatch otherwise — mirroring the _run selection. Add drain_relay_turn to the orchestrator.py 'modify' file entry and to Implementer task 6 (currently task 6 only branches _run). Add a test: an in-flight relay on a conversation build drains through run_conversation_turn (settles awaiting_manazer / blocked, never _persist_priprava_spec / _settle_phase_boundary).
- **[minor]** Task 3 adds 'paused' to _NOTIFY_STATUSES (pipeline_runner.py:44) so the existing _maybe_notify away-presence Telegram fires on the token-stop pause. But _maybe_notify (pipeline_runner.py:319) only runs on the FINAL settled state at the END of _run (line 251), AFTER the auto-chain + relay-drain loops. A token-stop pause set inside _run_build_round DOES bubble up as the returned settled state, so _maybe_notify sees status='paused' and (with the _NOTIFY_STATUSES change) will nudge — this part is sound. HOWEVER a plain Manazer-initiated 'pause' (apply_action pause at orchestrator.py:6506) also lands on status='paused'; adding 'paused' to _NOTIFY_STATUSES means a Manazer who paused the build themselves and is 'away' will now get a Telegram ping about their own pause. That is spurious noise (the Manazer just acted). The design's task-3 verify step doesn't cover this case.
  - **Oprava:** Distinguish the token-stop pause from a manual pause before nudging: e.g. gate the Telegram on the presence of the token_stop notification message (or a state flag / next_action marker set only by the token-stop seam), so _maybe_notify pings only for the automatic token-stop pause, not a Manazer's own 'pause' click. Add a test asserting a manual pause does NOT nudge while a token-stop pause does.
- **[minor]** The Telegram nudge body in _maybe_notify hardcodes the deep link '/vyvoj' (pipeline_runner.py:337: link = .../vyvoj). The spine retires /vyvoj to a redirect onto /riadiace-centrum. The link still resolves (via the App.tsx redirect the design adds), so it is not broken, but every away-Manazer Telegram nudge (awaiting/blocked/done AND the new token-stop paused) will point at the retired route and bounce through a redirect — a lingering reference to a CUT surface. The design's file list does not touch this line.
  - **Oprava:** Update pipeline_runner.py:337 to link to '/riadiace-centrum' (the spine page) as part of the routing switch (Implementer task 8), so the out-of-band nudge deep-links straight to the new Riadiace centrum instead of relying on the /vyvoj->/riadiace-centrum redirect.
